flask踩坑图鉴

2021年03月11日

用 Flask 时遇到的坑

这里记录几个离谱的大坑

关于请求时遇到308重定向

请求url时，如http://114514.1919810/auth/
如果使用的是url_prefix='auth'，路由写的是 ('/') [ 或是直接把路由写成了 ('/auth/') ] 并且实际请求的时候请求的
是 http://114514.1919810/auth 会导致重定向！
浏览器，postman什么的会跳转，但是response会获取到308!

关于extensions模块的使用

写flask相关的小程序时，常常会使用到db=SQLAlchemy(app),socketio=Socketio(app)之类常用的对象。
为了使程序更加具有可读性，可以将该部分拓展存放至extensions.py模块中，这里最好不要将app对象直接传入
扩展中，会导致循环引用，比较好的做法应该是在包的__init__.py文件中，导入extensions中的扩展对象，再
通过调用对象init__app(app)的方法来初始化对象，这样可以有效避免循环引用，同时可以增加文件可读性。

关于部署在服务器上

在本地测试时，如果有需要手写的文件路径，通常是使用/来表明文件结构，但是linux上的路径需要使用\ ，尤
其是写数据库地址的时候要记得更改，不然/会被识别为数据库文件名的一部分，无法连接正确的地址

关于socketio的使用

@socketio.on装饰器装饰的函数并不能被主动触发，必须要有提交才能触发，因此，不能通过死循环去进行与客户端
保持连接，会导致其他被@socketio.on装饰的函数无法被触发，可以通过广播的方式来实现像客户端发送消息，也可
利用房间的概念，将某一客户端加入单独的房间，再向该房间广播消息（话说轮询简单多了）

关于用户系统

如果要做用户系统，最好还是使用flask—login扩展，因为自己写安全性确实堪忧，而且繁琐的很….使用cookie的
话，cookie是不能跨域的，最好是通过token或者将前后端部署在同一个域名上，来进行登陆验证。关于具体的实现，现
还未掌握，前后端分离的方法的话可以参考一下PTA（想模拟提交是真的麻烦）。最后再记录一下，不要将密码的明文存入
数据库中，像sqlite3这种连登陆限制都没有的…安全性相当堪忧。最好是存加密过后的密文，用户发送请求的时候先使
用加密算法加密密码得到密文，再和数据库中的密文比对判断密码是否正确。切勿使用SHA1，md5等算法进行加密，因为算
法比较简单，很容易被正向比对后破解，建议使用慢哈希，flask可以使用werkzeug.security中的
generate_password_hash来进行加密，通过check_password_hash来进行比对，因为是慢哈希，所以不容易
被破解，顺带一提generate_password_hash的返回值是算法名$盐值$密文,反正怎么也比明文强多了。

关于装饰器

之前对登陆有需求，一个个函数去写又很麻烦，所以决定尝试装饰器写法（理所应当的又一堆bug，我吐了），
可能会遇到AssertionError: View function mapping is overwriting an existing endpoint function: wrapper
这样的错误，翻了下博客，果然又是装饰器出了毛病。
装饰器装饰后的函数，函数名wrapper.__name___会变成wrapper，当你联用多个装饰器的时候，就会报错，提示你
名字重了，和视图函数重名的错误信息相同，目前比较理想的解决方案是手动在装器下面设置wrapper.__name__为
func.__name__,或者也可以使用funcools.wrpps装饰下的wrapper函数，这里就不过多赘述了。因为还没学会
